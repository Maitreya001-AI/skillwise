## 0. 问题

生态里"把库去封装成技能"这个模式，只有数据层走得最远，通用层空白。设想一个通用"反编译器"元技能：输入任意库，按四原子拆解，按僵化成本公式判定该不该拆、拆成陈述态还是编译态。

三问：seek 现在做了什么？这个技能该长什么样？它在 WISE 里怎么加、该不该加？

---

## 1. seek 现在做什么

seek = write 的语料驱动版 + 存在性门（THEORY §10），原子构成 Control + Judgment + Capability。输入是**轨迹**——引擎跑任务留下的成功/失败执行迹。三条核心机制：

- **对比对为学习单元**：失败 × 邻近成功，定位"成功里有、失败里无"的行为；孤立失败无锚点。
- **批式归纳**：只对 ≥3 条共享签名的轨迹簇下笔，不对单条失败反应。
- **蒸馏模式不是路径**：问"哪个位点的缺口在反复出现"（事实→Σ / 即兴操作→Π / "好的样子"→φ / 非默认循环→γ），跑 write 物化，最后过存在性门（带/不带 delta on 留出集）认证。

一句话：seek 从**观察到的行为**里，反应式地发现引擎补不上的缺口。

---

## 2. 裁决：不是第五个字母

### WISE 是闭的——作为操作分法

四原子闭合是定理（§2，循环签名，抹一个区分损失一个失效名）。四元技能**没有**同等力度的闭合证明，§10 是"理论施加于自身"的构造。但 WISE 有干净的闭合读法——按**对技能的生命周期操作**分：

| 字母 | 操作 | 说明 |
|---|---|---|
| **E** evaluate | 量（measure） | 只读裁决 |
| **W** write | 创建（create-from-hypothesis） | 从人的缺口假设手写 |
| **S** seek | 创建（create-from-source） | 从语料/源发现 |
| **I** improve | 改（change） | 改已有的 |

create / measure / change 穷尽了对一个技能能做的操作。delete 塌回 evaluate 的裁决（删除测试不过 → 删），不单列。**WISE 作为操作分法是闭的。**

### 反编译器落在 create-from-source = seek 的操作

它拿一个库、产出一个技能，是 **create**，而且是 from-source（发现），不是 from-hypothesis（手写）。也就是它执行的**正是 seek 的操作**。跟 seek 只差两处：

- **源**：库（静态制品）vs 轨迹（行为迹）
- **读缺口的方式**：结构分解 vs 对比对归纳

**同一个操作，新的源。** 源不同 ≠ 顶层技能不同。seek 的定义就是"语料驱动的 write + 存在性门"，而库是一种语料（静态的）。反编译器是 seek 的一个**源适配器**，挂在 S 底下，不是 S 旁边的第五个。

> **自我修正**：上一轮我说过"姊妹而非特例"。缺口**源**确实不同、不可互替——这半对（僵化缺口对轨迹式 seek 不可见，因为 wrapper 压制了失败信号，直到接口崩才暴露）。但我把"源不同"错当成了"技能不同"。源不同不足以另立顶层技能；认证操作相同（都走存在性门）才是判据。

---

## 3. 正确形态：seek 的源适配器

把 seek 的"读缺口"这段重构成插槽，两个适配器：

- `from-traces`：对比对归纳（现有）
- `from-library`：结构四原子分解 + 僵化成本分诊（新，标 **L1 预测**）

共用尾巴不变：**四原子分类 → write → 存在性门（L2）**。

### from-library 适配器内部（三层）

- **前端（L1，预测）**：给任意库，拆成
  - K：硬编码的会变事实（字段语义 / 端点 / 限流）
  - Π：提供的操作
  - φ：内置的"对/允许"校验（schema 检查、错误分类）
  - γ：强制的循环（重试 / 退避 / 会话）

  再跑 `僵化成本 ∝ (K + J占比 × 变化率) ÷ Capability实现难度` 排序。**显式标注为预测**——它排优先级，不做认证。变化率从 git 历史 / 发版节奏 / issue churn 估。

- **body（write）**：物化——K 进 `references/` 懒加载，φ 写成声明式准绳，Π 声明存在，γ 写成 `done_when` 约束。Capability 那半**该留就留**（薄 wrapper，或不拆）。

- **门（L2，唯一认证）**：存在性门 = 带/不带 delta，vs SDK-wrapper 基线，留出任务**必须压在僵化面上**（接口变更 / 反爬边界 / 重组任务）。普通任务测不出僵化缺口——正常跑根本不崩。

- **负分支**：numpy 那种（Capability 实现难度高、K/J 占比低、变化率低）公式必须吐**负价值 → 不拆**。这是把"该不该触发"钉死的纪律，等价于库层面的删除/洗牌测试。

---

## 4. 为什么必须这样，而不是加字母

- 加 WISE+1 破坏一个干净的操作分法（把推导出的集合变成杂货铺）。
- 重复 write + gate 机器。
- 开坏先例：下个人为"从 git 历史 seek""从 issue tracker seek"加 WISE+2、+3。
- 把变化性推到源适配器层，正是四原子那种节制的一致做法，也守住宪章（删除测试当常设仲裁、无 registry、无 sprawl）。

**唯一让它够格独立成字母的情形**：需要一个不同的**认证操作**——"反编译有没有用"没法用带/不带 delta 回答。但它能（vs SDK-wrapper 基线，压在僵化面的留出任务上）。门共用 → 不需要新字母。哪天发现它认证走不了存在性门，再回来重估。

---

## 5. 好做吗

- **约 70% 是复用**：四原子分类前端（write/seek 共用）、write 物化机、存在性门（`shared/effect-gate.md`）都建好了。
- **新表面很小，且全是 Judgment（声明式）**：一张四原子分解判断表 + 僵化成本分诊 + 一条负分支。**技能本身：好做。**
- **真正的成本不在技能，在门**：认证"反编译后的库确实打赢 SDK-wrapper 基线"，要构造压僵化面的留出任务再跑带/不带。这正是项目**已经欠着的行为门**（关键技能的 behavioral delta 至今没跑）。反编译器不新增理论债，继承同一笔旧债。

**硬警告**：别让僵化成本那个数字冒充判决。它是 L1 预测器，只排队；认证权只在 L2。让公式的数替代 delta，就是拿 L1 的可靠度出 L2 的结论——违 §7。

---

## 6. 最后的实话：现在未必该建

反编译器是**发现**的放大器。项目真正卡住的不是"能不能发现更多缺口"，是"能不能在 L2 认证任何一个"（关键技能的行为门至今没跑）。**发现相对认证是廉价的。**

在门还没跑起来之前先造新发现前端，是在优化**非约束**的那一维。

**次序**：先把存在性门在一个真技能上跑通，`from-library` 适配器再上。

---

## 附：一句话版本

反编译器不是 WISE 的第五个字母，是 **seek 的 `from-library` 源适配器**（现有的是 `from-traces`）；共用四原子分类 → write → 存在性门；僵化成本公式住在适配器里当 L1 分诊器。好做（70% 复用），但**先跑通门，再建它**。
