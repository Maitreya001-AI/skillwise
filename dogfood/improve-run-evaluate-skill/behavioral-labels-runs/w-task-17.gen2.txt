- Service names use `PascalCase`, end with `Service` (e.g. `OrderService`), and match the proto filename/package (e.g. `order.proto` → `package myorg.order.v1;`).
- Package names are fully lowercase, dot-delimited, and encode API major version as the final component (e.g. `myorg.billing.invoices.v1`); no version number appears anywhere else (not in service, method, or message names).
- RPC method names are `PascalCase` verbs or verb phrases describing the action (`GetInvoice`, `ListInvoices`, `CreateInvoice`, `BatchGetInvoices`) — never nouns alone, never prefixed with `Rpc` or the service name.
- Method names use the standard verb vocabulary consistently across all services: `Get` (single, must exist), `List` (paginated collection, returns `NextPageToken`), `Create`, `Update`, `Delete`, `BatchGet`/`BatchCreate`/`BatchUpdate`/`BatchDelete` for batch variants — no synonyms (`Fetch`, `Remove`, `Add`, `Retrieve` are rejected).
- Every RPC has exactly one request message and one response message, each named `<MethodName>Request` and `<MethodName>Response` (e.g. `GetInvoiceRequest`/`GetInvoiceResponse`), never shared or reused across methods even when fields are identical.
- `List` methods' request messages include `page_size` (`int32`) and `page_token` (`string`); response messages include `next_page_token` (`string`) and a repeated field named after the plural resource, not `items` or `results`.
- Resource fields that reference another resource's identity are named `<resource>_id` or, for full resource paths, `name` (following `resources/{id}` REST-style path convention) — never bare `id` on its own when ambiguity is possible across joined resources.
- All field names are `snake_case`, all message/enum type names are `PascalCase`, and enum value names are `SCREAMING_SNAKE_CASE` prefixed with the enum's name to avoid C++ scoping collisions (e.g. `enum Status { STATUS_UNSPECIFIED = 0; STATUS_ACTIVE = 1; }`).
- Every enum's zero value is named `<ENUM_NAME>_UNSPECIFIED` and documented as the invalid/default sentinel — no enum ships without an explicit unspecified zero value.
- Deprecated fields/methods/messages are never renamed or renumbered; they are marked with `[deprecated = true]` and a comment stating the replacement, and field/method numbers are never reused after removal.
