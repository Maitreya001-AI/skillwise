- Resource labels (the part after the type) use `snake_case`, never contain the provider or resource type itself (e.g. `aws_instance.web`, not `aws_instance.aws_instance_web`), and never repeat the resource type in a way that becomes redundant when read as `type.name` (e.g. `aws_s3_bucket.bucket` is rejected).
- Every resource label follows `<purpose>_<qualifier>` ordering with the most stable/generic term first and the most specific differentiator last (e.g. `aws_subnet.private_use1a`, not `aws_subnet.use1a_private`), so alphabetical listing and `-target` globs group related resources together.
- Singular resources use singular names; when `count` or `for_each` is used, the label stays singular and the index/key carries the plurality (e.g. `aws_instance.worker[count.index]`, not `aws_instance.workers`).
- Module names are `kebab-case` for the source directory/registry name and `snake_case` for the local call label, and both must state the thing being provisioned, not the team or ticket that requested it (`modules/eks-cluster` / `module "eks_cluster"`, not `modules/platform-team-infra`).
- No environment, region, or account name is hardcoded into a resource or module label when that value is available as a variable or workspace attribute — encode it via tags/naming variables, not by baking `prod` or `use1` into the Terraform identifier, except where the physical resource name itself (the string passed to the cloud API) requires global uniqueness.
- Every cloud-facing resource name (the `name`/`name_prefix` argument, not the Terraform label) is built from a shared naming module or `local.name_prefix` convention of the form `<org>-<system>-<env>-<resource>` (or the org's documented equivalent) so names are greppable and collide-free across accounts.
- Data sources are named for what they look up, not mirrored off the resource that consumes them (`data.aws_ami.ubuntu_2204`, not `data.aws_ami.this` or `data.aws_ami.web_ami_lookup`).
- Reserved/ambiguous labels — `this`, `main`, `default`, `test`, `foo` — are forbidden on any resource or module block that has more than one instance in the codebase or that another module/resource references by name.
- Variable and output names mirror the resource concept they expose (`output "eks_cluster_name"`, not `output "name"` or `output "output1"`) and never shadow a Terraform meta-argument or provider-reserved word (`count`, `for_each`, `providers`, `depends_on`).
- When importing an existing resource into state, the assigned Terraform label must match this grammar even if the cloud-side resource name predates the convention — the import PR is rejected if it preserves a legacy ad hoc name in the `.tf` label rather than remapping it (the cloud-side `name` argument may stay unchanged via `name_prefix`/lifecycle rules if renaming would be destructive).
