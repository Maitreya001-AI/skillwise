# Improve a Skill

This is a **control** skill (composite: control + judgment + capability): a gap-classification front-end in front of a ratchet-and-gate loop. The derivation is in the skillwise repo's `docs/THEORY.md` (§8 the statistical charter, §9 evolution, §10 the meta-skill mapping). On first activation also read `references/rubric.md` (the *diagnosis* rubric — which dimension is weakest), `references/ratchet-protocol.md`, and `references/failure-driven.md`.

The keep/revert decision is **not** re-implemented here. Every edit is judged by the **shared measurement gate** in `references/effect-gate.md` — but on its **improvement gate** (`delta_step`: the edited version vs the last accepted version), *not* the **existence gate** `evaluate-skill` uses (`delta_exist`: vs no-skill). These are different questions: `evaluate-skill` asks *should this skill exist?*; `improve-skill` asks *did this edit make it better?* Conflating them is the failure that makes "improve" change nothing — when the base model is already strong, no-skill ceilings and *no* version can beat it, so a loop keyed on `delta_exist` rejects every edit. The no-skill score stays a **floor/guard**, never the bar each round must clear.

> **Where the theory licenses this loop.** A control skill leaves *task* iteration to the engine, but the gate protocol below is concrete and non-skippable — the failure being prevented is the engine skipping its own gate (A′-1). That is not a special exemption: the loop's order — change → measure → keep/revert — is an **epistemic order**, cell 4 of THEORY §3 (you cannot select on a delta you have not measured), and its non-skippability is §4's compilation requirement on Control. An instance of the theorem, not a refinement of it.

## Phase A — Classify the target

Name the gap the target fills (Knowledge / Capability / Judgment / Control / composite) and its **tier** (scaffold vs production/library — this decides whether the effect layer is mandatory; see `references/effect-gate.md`). Classification selects the type-aware diagnosis weighting in `references/rubric.md`: workflow-clarity is N/A for Knowledge/Judgment skills; executable-specificity relaxes for Judgment. Run `evaluate-skill` once as the **entry check**: the existence floor (`delta_exist` vs no-skill) and the current structural findings.

If the skill ties or loses to no-skill, apply the gate's **sub-floor routing** before looping (one rule, shared — see effect-gate's "Sub-floor routing"): if `no_skill` already passes the typical tasks, the finding is **`nogap`** — the skill should not exist; retire it, do not run this loop. If the gap is real (`no_skill` fails) but the skill scores below it, that **negative transfer is the first thing to fix** — it is this loop's first repair target. If the skill is already strong, that is fine too — improvement is still measured each round against the skill's *own previous version*, not the floor.

## Phase B — Set up the harness

- Confirm the target holds a `SKILL.md` and the git tree is clean (the ratchet reverts via `git checkout`).
- Create a working dir with rolling memory: `learnings.md` (verified wins) and `dead-ends.md` (falsified edits, never retried).
- **Assemble the held-out set per `references/effect-gate.md`**: caller-supplied tasks, or draft 3 happy-path + 1 edge into `baseline.md` and confirm with the user before scoring — noting that a 4-task set is below the gate's power floor (`n_tasks ≥ 6`), so its verdicts are **indicative only**; extend the set to certify.
- **Run the gate's power check** and emit the `power` block (`resolution = 1/n_tasks`, `adequate`); an inadequate set that cannot be hardened is `unfit_test_set`, not a mandate to iterate anyway.
- **Reserve the confirmation slice (§8-4).** Hold back ~25% of the tasks (at least 2) as `confirm/` — invisible to every round; the loop iterates only on the working slice. The terminal `improved` verdict must be re-confirmed on this slice (see done_when). **Rounds on one set are capped at 10**: at the cap, rotate the set or stop.
- **Calibrate for headroom (mandatory, before any round).** Compute the once-only `no_skill` baseline *and* the current skill's scores, and run the gate's **headroom check** — for the improvement loop it keys on the **current skill**: fit iff some task scores below max (an edit could raise it) *or* below `no_skill` (negative transfer to repair); variance across tasks is preferred but not required. Emit the `{headroom, variance, fit}` discrimination report. `no_skill` is only the floor; it may sit at ceiling without making the set unfit (improvement is measured vs the previous version, not vs no-skill). The set is `unfit_test_set` **only** when the current skill is already maxed on every task with no negative transfer → that is `already_optimal`, not a loop to run. Harden the fixtures (cases that expose the gap) before iterating. Weak tests inflate; a set with no headroom for the question hides the answer.

## Phase C — The loop

Each round is propose candidates → pick the best vs the current version → independent re-score through the gate's **improvement** gate → forced keep/revert.

1. **Diagnose** the single weakest dimension via `references/rubric.md`. If you have execution traces with outcome labels, switch the diagnosis source to failure-driven mode (`references/failure-driven.md`): pair each failure with a neighboring success and extract the missing-behavior signature. Read `learnings.md` + `dead-ends.md` first; never re-walk a falsified edit. **One defect per edit.**
2. **Generate ≥3 diverse candidate edits** that all target the *single diagnosed defect* — alternative fixes of it, spread across mechanism, not three different defects (that would violate "one defect per edit"). Not a single bet: the lone-diagnosis guess is often wrong (it was, in this skill's own dogfood run). Each respects the **bounded step: ≤30 changed lines within the skill directory** ([SkillOpt](https://arxiv.org/abs/2605.23904)'s textual learning rate). The scope is the whole skill directory — `SKILL.md`, `scripts/`, `references/` — because a Capability defect is legitimately repaired by adding or fixing a named primitive in `scripts/` (§4 requires Capability gaps in compiled form; a SKILL.md-only scope would forbid a class of correct fixes).
3. **Screen the candidates** with one fast, cheap, *non-blind* scoring pass against the current version to pick the single most promising — this selection heuristic is **not** the gate and does not decide KEEP. Run the screening on **throwaway copies** so the working tree stays clean; then **snapshot the clean current-accepted tree via commit** (the ratchet's revert baseline, per `ratchet-protocol.md` — snapshot *before* the edit) and apply only the chosen candidate.
4. **Re-score through the shared gate (authoritative).** Run `references/effect-gate.md` on the chosen edit: held-out **edited vs previous-accepted** (`delta_step`), reusing the once-computed per-task `no_skill` only for the floor check. **Re-run the reference (previous-accepted) condition exactly 3×** to size the band (`noise_band = 2 × sample standard deviation` of the three aggregates; other conditions ≥2×); per-task scores are **medians across re-runs**; judges blind and **decorrelated** (different models or information-isolated), median of ≥2. Only this run's `delta_step` decides KEEP. It yields the `gate` object — read `gate_pass`, and from its `effect` block `delta_step` / `delta_exist` / `floor_ok` / `regression_count` / `safety_regression` / `cost_delta`. This run is compiled — do not hand-assemble it: `python scripts/gate_runner.py run tasks.yaml` with `question: improvement` + `prev_accepted` (its docstring has the schema) runs the arms in isolated sandboxes, keeps the `confirm:` slice physically out of every round (re-check it once, at the terminal `improved`, via `--confirm-slice`), stores reusable condition scores (`no_skill_baseline:`), and records per-run cost.
5. **Ratchet** (`references/ratchet-protocol.md`) on the gate's output. Default is revert.

## The decision (delegated to the shared gate's improvement gate)

Keep/revert reads the shared gate, it is not re-derived here:

- **Keep** iff `delta_step > noise_band` **and** no task whose *median across re-runs* drops below the previous accepted version **and** `with_edited ≥ no_skill` on every task (medians — floor intact) **and** no safety regression **and** `evaluate-skill`'s Tier 1 introduced no new blocking structural finding.
- **Lateral keep** (`delta_step` within the noise band — with no reproduced per-task regression vs previous, floor intact, and safety-clean) only if the edit *removes* a Tier-1 blocking finding; otherwise a within-noise change is a revert.
- **Revert** on any reproduced per-task regression vs the previous version, any floor breach (median `with_edited < no_skill` on a task), any `safety_regression`, or a new blocking structural finding — the gate's fatal, non-waivable outcomes. A skill worse than its previous version or below the floor, or less safe, never lands, whatever the average.
- **`unfit_test_set`** is neither keep nor revert — **halt**, harden the held-out set (Phase B), then resume. Never keep on a set that cannot measure a gain.

After deciding: KEEP → distil the win into `learnings.md` and make the edited scores the new previous-accepted; REVERT → distil the failure into `dead-ends.md` so it is never retried. Either way, log the round (including `cost_delta`) to `experiments.tsv`.

## Plateau-break

When N rounds pass with no KEEP, do one larger exploratory rewrite that may exceed the step budget (branch first; consult `dead-ends.md`; the gate's fatals still apply). Clears the improvement gate → adopt and reset the counter; else return to the snapshot. One per skill. (A plateau cannot be broken by a bigger edit when the ceiling lives in the *test set* — that is `unfit_test_set`, handled in Phase B, not here.)

## done_when — terminate into one honest state, named explicitly

The user asked for improvement, so a near-zero-change run must resolve to a real verdict, never silently to "done":

- **`improved`** — ≥1 edit kept, each beating the previous version beyond the noise band, **and the surviving version re-confirmed on the confirmation slice** (`delta ≥ 0` vs the entry version on `confirm/`, which no round ever saw). Report the cumulative `delta_step` gain. If the confirmation slice fails, report **`improved (unconfirmed)`** instead — the working-set gain may be adaptive overfitting (§8-4) — and recommend rotating to a fresh set before trusting it.
- **`already_optimal`** — the held-out set **ran**, the **surviving version is floor-clean** (`≥ no_skill` on every task), but it can't be measurably and safely improved: every tried edit either fell within/under the noise band (no fatal) or triggered a fatal and reverted, or the headroom check found the current skill already maxed on every task (`unfit_test_set`, no negative transfer). This is "as good as *this set* can show" — pair it with the caveat that a harder set may still surface faults, and recommend hardening before concluding the skill is globally optimal.
- **`cannot_measure`** — the **harness itself** couldn't produce scores: no executor, no obtainable held-out set (the gate's `static_only`), or scoring errored out. The null result is the harness's fault, not the skill's: do **not** report success. Escalate with the remedy — supply/repair the held-out set.

These are mutually exclusive: `unfit_test_set` (current skill maxed, no negative transfer) routes to `already_optimal` (consistent with `references/effect-gate.md` and the ratchet), **not** `cannot_measure` — which is reserved for a genuinely unrunnable harness.

Stop also on the per-set round cap (default 10 — rotate the set or stop), max rounds, or user stop. A version carrying negative transfer (floor breach) or a safety regression never counts as done — fall back to the last clean KEEP. **If no clean KEEP exists and the surviving (entry) version itself breaches the floor (no KEEP ⇒ the tree was reverted to entry) — it loses to no-skill and no repair landed — the run did *not* reach a done state: the repair failed. Follow the gate's sub-floor routing: escalate to `seek-skill` for a structural rethink (or supply a stronger fix); never report `already_optimal` for a sub-floor skill.** A run the user asked to improve that ends with ~0 change must therefore resolve to one of: `improved (unconfirmed)` (kept, but not confirmed), `already_optimal` (floor-clean, no measurable gain), `cannot_measure` (harness unrunnable), or the failed-repair escalation above — never an unqualified "done".

## What this is not

Generation (0→1) is `seek-skill` (from a corpus) or `write-skill` (by hand). A one-shot verdict with no loop is `evaluate-skill`. This skill evolves **one skill directory** — `SKILL.md` plus its `scripts/` and `references/` — behind the gate; it does not rename `name`, mutate two skills at once, or exceed the bounded step. Repairing a Capability defect by shipping a primitive in `scripts/` is in scope (§4); rewriting the world outside the skill's directory is not.
