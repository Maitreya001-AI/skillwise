0前提

引擎是免费的

The premise — the engine is free

一台现代基座 agent 已经自带通用 symbol 层推理引擎:读 schema、写代码、把意图映射成操作、从残缺输入恢复、编排多步工作——外加截止训练时的通用知识。称这台引擎的存在为命题 A。它是免费的。

一个 skill 挣得存在的唯一方式,是补上这台免费引擎在某个具体任务上缺的东西。于是整个设计问题坍缩成一句:

它,到底能缺什么?

1分类

缺口的种类,就是 skill 的种类

Classification — the kinds of gap

把引擎已有的列全,缺口只剩四种。MECE,且对齐知识层 / symbol 层。

公平起见,最强的相反观点值得一听。一篇综述主张"操作过程"本身值得外化,因为很多 agent 错误是过程级的(跳步、错序、过早终止)[Externalization 综述]。调和很精确:那篇综述把技能拆成三件——操作过程、决策启发、规范约束;后两者正是本文的判断与控制原子。分歧只在第一件:把字面步骤序列冻死会引发 inertia,而把停止条件与决策启发编码下来,既抓住真实的过程知识、又不焊死顺序。过程不是一种类型;"过程"里真正算缺口的部分,是判断和控制。

真实 skill 大多是复合体:

领域 / 本体型
知识+能力+判断
抽取 / 管线型
控制+判断+能力

复合体一分解,human checkpoint 该插哪就显形——它落在判断分量与控制 / 能力分量的交界上。

2写什么

描述世界,把"怎么做"留给引擎

Content — describe the world

一份好 skill 只描述三样,然后闭嘴:agent 操作的世界(概念/类型/关系)、这个世界里什么算对(约束/不变式/语义信号)、用户大概想要什么(词汇/场景)。流程不是忘了写——那是引擎的活,它按上下文适配,比任何冻死的脚本都强。把 skill 写成"正确输出的属性"、而非"产出的过程",也是公认的实践 [Agent Skills best practices]。

面向领域复合体的一套默认骨架,从"agent 用一个元素时走的推断链路"反推而来:Vocabulary + Purpose(该用这类元素吗)、Contrast(易混时才要)、Structure(字段取什么)、Relationships(怎么挂)、Rendering(可视时才要)、Rules(硬不变式)、Heuristics(软经验)。

Rules 段有第二条命:每条硬不变式既是该做成 primitive 的候选,也是出口该检查的候选。Heuristics 段不生成任何机械物——那里是判断,判断归引擎。

3隐藏缺口

承重句拆成两个命题,bug 住在裂缝里

The hidden gap — primitives & a semantic exit

"你描述知识,引擎补实现"——偷偷打包了两个命题:A 引擎存在(有通用推理能力,真)与 B 原语现成(要操作的领域原语已摆在那,常假)。

B 失效时,引擎有能力、没素材,当场即兴造一个用完即逝、无人审查的私有原语——这层就是 bug 的住所。典型:规模一大引擎临时写脚本批量生成,函数按位置填参,标签值掉进备注字段;skill 看不见脚本、位置参数没名字提醒、校验器只问"格式合不合法"——三道关卡同时沉默。轨迹原样编译进 skill 的抽取器正会复制这种坑 [AXIS]。

4正确性

一把能跑的尺 ——对 = 恰好补上缺口

Correctness — a ruler you can run

不漏、不溢、补在对的层、出口能验。五个测试各抓一种错并报出错名。

  1. 删除测试 —— 有没有缺口。拿掉 skill 还过 → 它在供应命题 A,本就不该存在(纯溢填)。先问该不该有,再问对不对。
  2. 即兴测试 —— 漏没漏。跑典型任务,数引擎为缺口即兴写了多少。>0 = 漏填。
  3. 洗牌测试 —— 溢没溢。primitive 打乱仍可完成=词汇;只在你的次序下成立=过程焊死。
  4. 惯性-成本测试 —— 溢的另一面。已能过的任务 with/without 对比,更长更贵无增益 → inertia。
  5. 出口测试 —— 可不可保证。有没有对照"正确长什么样"的语义检查?没有 → 判不对。

复合体多一条接缝测试:human checkpoint 是否落在判断与控制 / 能力的交界?每型还各有特征失败与出口闸门(知识=新鲜度;能力=测试+合理性;判断=plausible;控制=done_when)。

别靠读 skill 来筛它。文本顺滑度已与真实效用脱钩:让一个 LLM 评委判"两个 skill 哪个更好",无引导时只有 46.4% 命中、劣于掷硬币,且常挑那个读起来更顺却更差的 [SkillLens]。所以——把 skill 当成一种干预、对照无-skill 基线来判 [SkillGen];取 ≥2 个独立评委的中位。三个读得出、又确实预测效用的信号:是否编码失败机制(如果 X 则 Y 否则 Z)、指令是否可执行而非含糊、是否有专门的高风险黑名单。这些要类型化地用——对知识/判断型,"有工作流"不是优点。

5进化

门后单调变好

Evolution — monotone improvement behind a gate

要让 skill 随经验变好且不退化、不漂移,控制门优先于一切聪明的归纳法。

门做什么出处
批量归纳对多样轨迹池整体归纳,而非逐条反应(后者过拟合轨迹局部教训)Trace2Skill
验证门每次写入须在 held-out 上严格更优,否则丢弃 —— 把无条件自编辑变成 propose-and-testSkillOpt
有界步长文本学习率,禁单步破坏性重写SkillOpt
负知识缓冲被拒编辑连同它造成的掉分一起记下,不再重犯SkillOpt
经验饮食别用全失败池;成功/失败配比分领域调SkillLens
角色分离抽取≠执行≠评估;信息隔离的验证器无真值也能给可靠信号CoEvoSkills
安全损失拒答率/攻击成功率/越权判断进同一个门;skill 文件是真实攻击面Skill-Inject
破局停滞时允许一次越预算的探索性重写,过门才留—
6合流

一条轴 ——留给引擎多少

Confluence — one axis

分类看它补哪个缺口 → 写就是把缺口在对的层填满、把引擎已有的削尽 → 隐藏缺口靠 primitive 入口 + 语义出口兜住 → 正确性用五测+接缝判定 → 进化让它在门后单调爬升。一条链,一条轴。

7四件套

四个处理 skill 的 skill,从链尾掉出来

Four meta-skills, one kernel

skill它是链的哪一截类型
evaluate§4 那把尺本身(内核)判断+能力
write§4 把尺倒过来跑(手写)判断+知识
seekwrite 由语料驱动(从经验发现)控制+判断+能力
improve§5 把尺放进带门的循环控制+判断+能力

自指约束:这四个 skill 本身都遵守它们所编码的理论——都声明式、不写 Step 1/2/3;evaluate 配机械检查 primitive(入口)却把判决交给语义读(出口);improve / seek 只声明 done_when + 角色分离 + 门,把任务迭代留给引擎。

8修订

两条让规则不自相矛盾的修订

Two refinements

修订一

洗牌测试豁免"测量协议顺序"

验证门的核心循环"改 → 测 → 保留/回滚",顺序是认知性的——你无法在不"改完再测"的前提下设门。区分:任务编排顺序(钉死 → inertia,禁)vs 测量协议顺序(改后才能测,合法)。后者不算溢填。

修订二

"迭代留给引擎"对控制型加限定

agent 不会可靠地自我设门——这正是自生成 skill 会拉低表现的原因。所以:任务迭代留给引擎,但门协议必须具体、不可跳过。这正是修订一的推论:门协议就是那条合法的测量协议顺序。

9两个前端

抽取(寻)与失败驱动

Two front-ends — extraction & failure-driven evolution

§5 管"改一个已存在的 skill"。两个前端决定 skill 从哪来。生命周期三段:经验生成 → 技能抽取 → 技能消费 [SkillLens]。

自进化需要什么信息(最小集):① 完整轨迹(含引擎即兴造的私有原语,§3 的缺口住那);② 每条轨迹的结果信号;③ 对比——失败配一个最近邻成功(孤立失败无锚点);④ 一个与挖掘池不相交的 held-out 集(让它成为优化而非蒸馏);⑤ 当前 skill 状态(改而非重发现);⑥ 被拒编辑缓冲;⑦ 语义验证器;⑧ 安全信号;⑨ 经验饮食旋钮。

抽取(寻)= write 由语料驱动 + 分类前端 + held-out 门。必避的坑:把轨迹(动作序列)原样编译进 skill = 溢填。抽取要蒸馏的是模式(复发的是哪个缺口),不是路径。循环:批量归纳挖复发模式 [Trace2Skill] → 先分类缺口(缺事实→知识/误造操作→能力命名原语/判错好→判断/非默认循环→控制) → 按型物化 → 注入后 with/without 过门 [MUSE-Autoskill] → 去重保护、越种子语料的扩展也走同一道门 [SkillX]。

失败驱动 = improve 把诊断源换成对比失败对。单位是对比对(失败 × 最近邻成功),不是孤立失败:定位"成功里有、失败里没有"的那个行为差 [SkillGen]。抽签名 触发 → 征兆 → 成功走了而失败没走的分支,喂进同一道棘轮与门。若某失败簇根本没有邻近成功 → 该 skill 缺新能力,转抽取,而非编辑 [CODESKILL]。

三个自进化失败模式正当化了这些门:记忆腐烂(早期错编辑被当真值回放 → held-out 用新鲜任务)、顺序过拟合(逐条反应过拟合 → 按失败簇 ≥3 批量归纳,绝不 n=1)、误进化(累积补丁拉低安全对齐 → 安全门每次都跑)。无门的自我修改不是进化,是受控的退化。

一句话

分类看它补哪个缺口,写是把缺口在对的层填满、把引擎已有的削尽,正确性在出口处验,进化在门后单调爬升。其余关于 skill 的"好",都是这一句的影子。
